Queue 排队机制：方案对比

当会话处于 Busy 状态时，用户发送新消息该怎么处理？

方案 A：客户端本地队列
🤖 正在处理上一个请求...
⏳
排队中 (1)
"帮我重构这个函数" — 等待发送
✅ 消息安全保存，空闲后自动发送
工作方式：
  1. 用户在 Busy 时发消息 → 存入本地队列
  2. UI 显示 "⏳ 排队中 (N)" 徽章
  3. 收到 SSE session.idle → 自动出队发送
  4. 队列清空 → 徽章消失
方案 B：禁发 + 提示
🤖 正在处理上一个请求...
⛔ AI 正在回复中，请等待完成后再发送
帮我重构这个函数
发送
工作方式：
  1. Busy 时发送按钮/输入栏变灰
  2. 显示 "AI 正在回复中" 提示
  3. 收到 session.idle → 恢复可发送
  4. 用户自行等待后发送

权衡对比

方案 A 优势

  • 用户体验更流畅，不打断思考
  • 消息不会丢失
  • 更接近 Web 端体验

方案 A 劣势

  • 实现复杂（队列管理、状态持久化、边界情况）
  • 如果用户排队 3 条消息，服务端可能无法按序处理
  • 需要处理横竖屏切换、进程重启等场景

方案 B 优势

  • 实现极简（只需一个 isBusy 检查）
  • 不会出现乱序问题
  • 与 Shell 模式现有行为一致

方案 B 劣势

  • 用户必须等待，体验略差
  • 长回复时用户可能等很久
